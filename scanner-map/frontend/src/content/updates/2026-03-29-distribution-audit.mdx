# Release Summary

This release closes the gap between the published Windows installer artifact and the actual delivery pipeline used by the app.

- `fix`: added a backend-owned Windows download route that resolves the current installer artifact and streams it with forced attachment headers.
- `fix`: normalized explainer CTAs so every Windows download path points at the same backend route.
- `sec`: added explicit attachment and MIME handling for executable delivery.
- `chore`: documented the current packaging chain as PyInstaller onedir plus Inno Setup, replacing older mixed assumptions.

# What Changed

The backend now serves the Windows installer from a single route instead of relying on mixed GitHub-release links, relative file references, or browser heuristics. The route prefers the latest installer in `dist_installer`, falls back to the local explainer copy when needed, and finally falls back to the raw bundled executable.

```python
@app.api_route("/download/windows", methods=["GET", "HEAD"])
def download_windows_installer(request: Request):
    download_path = resolve_windows_download_path()
    headers = {
        "Content-Disposition": 'attachment; filename="Invincible_Inc_Windows_Sovereign.exe"',
        "Content-Length": str(download_path.stat().st_size),
        "Cache-Control": "no-store",
    }
    media_type = "application/vnd.microsoft.portable-executable"
```

The backend streams the file in 64 KB chunks instead of loading the whole artifact into memory.

```python
def iter_file_chunks(path: Path, chunk_size: int = 64 * 1024):
    with path.open("rb") as handle:
        while True:
            chunk = handle.read(chunk_size)
            if not chunk:
                break
            yield chunk
```

The explainer page now binds all Windows CTAs to the same trusted-click flow and provides visible download state feedback.

```html
<button
  id="hero-download-btn"
  onclick="return triggerWindowsDownload(event, this)"
  class="btn btn-primary"
>
  ⬇ Download for Windows
</button>
```

```js
async function triggerWindowsDownload(event, btn) {
  if (event) event.preventDefault()
  const target = btn || (event && event.currentTarget)
  const downloadUrl = target.getAttribute('href') || '/download/windows'
  setDownloadButtonState(target, 'isDownloading', 'Initializing...')
  const headResponse = await fetch(downloadUrl, {
    method: 'HEAD',
    cache: 'no-store',
    credentials: 'same-origin',
  })
  if (!headResponse.ok) throw new Error('Download route returned ' + headResponse.status)
  ensureDownloadFrame().src = downloadUrl + '?t=' + Date.now()
}
```

# Packaging Notes

The current Windows packaging chain is:

1. PyInstaller onedir bundle in `scanner-map/dist/InvincibleInc`
2. `_internal` contents directory for collected runtime files
3. root-level `python312.dll` mirror for Windows loader compatibility
4. Inno Setup installer output in `scanner-map/dist_installer/InvincibleInc_Setup_v1.1.exe`

The project is currently shipping from PyInstaller plus Inno Setup, not Nuitka. Historical references to Nuitka-style include flags were not used by the final installer that passed verification in this cycle.
